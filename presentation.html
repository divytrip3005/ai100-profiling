QAic MatMul Performance

Kimi LLM — Prefill & Decode Profiling on Cloud AI 100

Understanding where time goes and how to recover it

🎯 WHAT ARE WE MEASURING?

A single Matrix Multiplication from the Kimi LLM.

Every attention layer and FFN layer is built on MatMul. If MatMul is slow, the whole model is slow.

Input: [S × 7168] tokens
Weights: [7168 × 18432] ~98.5 MB MXFP6
Hardware: 16 NSP cores Cloud AI 100

⚙️ HARDWARE

16 NSP Cores 8 MB VTCM / core ~130 GB/s DDR BW MXFP6 Weights DBG SDK v1.22.0

📋 THREE TEST SCENARIOS

seq1024 1024 14.6 MB Long Prefill

💡 Weights are always 98.5 MB regardless of seqlen. The question is how efficiently we reuse those weights across tokens.

What Is the Hardware Actually Doing?

Each of the 16 NSP cores runs 5 parallel threads — every thread has a dedicated job

🧠 NSP CORE ANATOMY (×16)

📡
DMAIssue
Issues DDR→VTCM transfer commands to the DMA engine
🚛
DMA
Physically moves weight tiles from DDR into on-chip VTCM
🔧
HVX
Decompresses MXFP6 → float16, handles layout conversions
📢
HVX_DMA
Broadcasts weight tiles from this core's VTCM to 3 sibling cores
⚡
HMX
The matrix engine. Performs actual float16 MatMul. This is the only thread doing "real" compute.
Total active threads: 16 cores × 5 threads = 80 threads running simultaneously

Only 5% of time is real compute. The HMX engine does its work in ~10 ns per tile but waits ~55,000 ns before each tile is ready — a 5,500× stall-to-work ratio. This is because constantPrefetchAmount = 0: the compiler issues no look-ahead DMA. Enabling 2-tile prefetch with double buffering could recover most of this idle time.

MatMul Operation Sequence — Serialized View

What each thread does, in order, for every weight tile — repeated 344 times (h7168 × seq1, 1 device)

# Thread Operation What it does (simple)
1
 DMA aiccopytovtcm
Copy one weight tile from DDR (slow, off-chip) into VTCM (fast, on-chip)
~24 µs  ·  loads ~175 KB of mxfp6 weights per tile
⏳ sync DMAIssue — HVX waits until DMA confirms the tile is fully in VTCM
2
 HVX blockdequantize_mxfp6
Decompress weight tile from mxfp6 (6-bit) back to float16 (16-bit)
~11 µs  ·  ~2.67× size expansion, stays in VTCM
3
 HVX aicconverttod32
Reformat the input token from flat float16 layout → D32 layout
~0.3 µs  ·  D32 is the internal format HMX understands
4
 HVX aicmulticastvtcm
Broadcast the formatted input token to all 16 cores via on-chip VTCM bus
~7 µs  ·  so every core can multiply against its assigned weight slice
⏳ sync HVX — HMX waits until HVX finishes broadcasting the input before it can start computing
5
 HMX ⚡ aicconvolutiond32
Actual matrix multiply. Input token × weight tile → partial output chunk
~6 µs  ·  this is the only step doing real math — everything else is preparation
⏳ sync HMX — HVX waits ~29 µs for HMX to finish before it can reformat and pass the result downstream
6
 HVX aicconvertfromd32
Convert the partial result from D32 layout back to flat float16
~0.3 µs  ·  output is now in standard format
7
 DMA → next tile
Go back to step 1 for the next weight tile. Repeat 344 times total.
344 tiles × ~55 µs/tile = ~19,000 µs total wall-clock for h7168 × seq1 on 1 device
8
 HVX aicoutputsemaphoreinc
Signal host that all tiles are done and the final output row is ready
one-time at the end  ·  assembles partial results from all 16 cores
⏱
Per-tile time budget:   DMA load ~24 µs → dequant ~11 µs → multicast ~7 µs → HMX compute ~6 µs

Actual compute is only 6 µs out of ~55 µs per tile — 11% of the time. The other 89% is moving and reformatting data.
💡
Why 4 devices is faster at seq=1:

Each device gets ¼ of the output channels → only ~86 tiles instead of 344. Fewer tiles = fewer DMA loads + fewer sync stall cycles. The 4× speedup comes from reducing tile count, not from adding compute power.
